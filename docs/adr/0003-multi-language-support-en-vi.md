## Context

The PRD's target users are "individual users and students, primarily in Vietnam" (PRD line 32), but the app ships with English-only UI text. A code audit found the codebase is in a **mixed, non-compliant state** for localization:

- `app/src/main/res/values/strings.xml` exists with 21 externalized strings, referenced via `stringResource()` in 10 files.
- However, **~48 additional UI strings are hardcoded as Kotlin literals** directly in `Text(...)` and `contentDescription = ...` calls across `DocumentViewerScreen`, `EditScreen`, `DocumentListScreen`, `ScannerScreen`, `SettingsScreen`, `RenameDialog`, `DeleteConfirmDialog`, and `ManualCropScreen`. One of these (`SettingsScreen.kt:98`, `"Có gì mới"`) is *already hardcoded Vietnamese* mixed into an otherwise-English screen — direct evidence the app cannot currently be localized without a refactor first.
- There is no locale-switching mechanism, no `values-vi/` resource directory, no `android:localeConfig`, and no `androidx.appcompat` dependency (confirmed via `libs.versions.toml` and `app/build.gradle.kts`).
- `SettingsScreen` renders release notes from `app/src/main/assets/release_notes.yaml`, which is **auto-generated by CI** from git log (`.github/workflows/release.yml`, per the commit history excluding test/coverage commits). This content sits outside the Android resource system entirely.
- `formatBytes()` in `SettingsScreen.kt` uses `"%.1f".format(...)` with the platform default locale — under `vi-VN` this renders a comma decimal separator (e.g. `1,5 GB`) next to an untranslated English unit abbreviation, which is an inconsistent (if minor) locale bug once `vi` is activated.

This ADR decides: (1) how to externalize the remaining hardcoded strings, (2) how the app resolves English vs. Vietnamese at runtime, (3) how to treat non-resource content (release notes, numeric formatting), and (4) how to prevent regression (untranslated strings shipping silently).

---

## Decision Drivers

- **PRD alignment**: primary user base is in Vietnam; Vietnamese must be a first-class, complete translation — not an afterthought.
- **Simplicity** (carried over from ADR-0002): solo developer → avoid a locale-switching implementation that requires maintaining two OS-version-specific code paths.
- **Zero new dependencies** (tech-strategy.md): any new library must be explicitly justified; default answer is no.
- **Fully offline** (PRD NFR-3): no server-side or on-device machine translation is acceptable.
- **Two Hats Rule** (code-quality.md): string extraction is a pure refactor (no behavior change) and must not be mixed into the same commit as the Vietnamese-translation feature work.
- **Regression safety**: once two locales exist, a string added to English and forgotten in Vietnamese must be caught before merge, not discovered by a Vietnamese user in production.

---

## Decision A — String Externalization Strategy

### Option A1: Extract all hardcoded strings to `strings.xml` in one refactor pass

Move all ~48 literals (including interpolated ones like `"Page ${pageIndex + 1}"` → `<string name="page_number">Page %1$d</string>`) into `values/strings.xml` before any translation work starts.

| Pros | Cons |
|------|------|
| Single clean baseline; lint's `HardcodedText`/`MissingTranslation` checks become meaningful immediately | One large diff touching 8 files |
| Matches Two Hats Rule — pure refactor, tests assert identical (English) output | Requires re-running/adjusting `EditScreenTest.kt` / `DocumentListScreenTest.kt` assertions that match on literal text |
| Unblocks Decision B/C independently of implementation order | |

### Option A2: Extract incrementally, one screen per PR

| Pros | Cons |
|------|------|
| Smaller, easier-to-review diffs | App is in a half-translated state for multiple PRs; lint warnings noisy/misleading mid-migration |
| | PRD open question stays unresolved longer |

### Option A3: Leave literals hardcoded, translate only the existing 21 `strings.xml` entries

| Pros | Cons |
|------|------|
| No refactor needed | **Not viable** — a hardcoded Kotlin string literal cannot be resolved per-locale; ~70% of visible UI text would stay English regardless of device language |

### Decision: **A1**

**Rationale:** The PRD explicitly targets Vietnamese users; a partially-translated app (A2/A3) actively contradicts that goal. A1 is a single Hat-1 refactor commit — mechanical, low-risk, and it is the correct prerequisite regardless of which locale-switching option is chosen in Decision B.

## Decision B — Locale Resolution / Switching Mechanism

### Option B1: System-locale-only (rely on Android's built-in resource qualifier resolution)

Ship `values/strings.xml` (English, default) + `values-vi/strings.xml` (Vietnamese). The OS automatically serves `values-vi/` when the device language is Vietnamese, and falls back to `values/` otherwise. No runtime code.

| Pros | Cons |
|------|------|
| Zero new dependency, zero new code | User cannot force Vietnamese if their device is set to a third language (e.g. French) — always falls back to English |
| Standard, well-understood Android behavior | No in-app way to preview/switch language without changing device-wide settings |
| Fully testable by changing the emulator/device locale | |

### Option B2: In-app override via platform `LocaleManager` (API 33+) + manual `Configuration`/`attachBaseContext` shim (API 26–32)

Persist a user-chosen language in `SharedPreferences`; apply it via `context.getSystemService(LocaleManager::class.java).applicationLocales` on API 33+, and via a hand-rolled `Configuration` override in `Application.attachBaseContext` / each `Activity` on API 26–32 (min SDK is 26, so ~2 major OS versions need the manual path).

| Pros | Cons |
|------|------|
| Zero new dependency (uses only `androidx.core`, already present) | Two divergent code paths to write, test, and maintain (pre-33 vs. 33+) — direct conflict with "Simplicity: solo developer" driver |
| User has full control over displayed language, independent of device locale | Manual `Configuration` overrides are a well-known source of subtle bugs (locale not re-applied after process death, `Activity.recreate()` timing, resources cached with stale locale) |

### Option B3: `androidx.appcompat` `AppCompatDelegate.setApplicationLocales()` with `autoStoreLocales` (AppCompat 1.6+)

Add `androidx.appcompat:appcompat` and a manifest meta-data flag; AppCompat's `ActivityLifecycleCallbacks` auto-applies and persists the chosen locale for **any** Activity (no need to extend `AppCompatActivity`, so the existing Compose/Material3 theming is untouched).

| Pros | Cons |
|------|------|
| ~10 lines of app code vs. B2's two hand-written code paths; Google-maintained and battle-tested across API 26+ uniformly | **New dependency** — violates tech-strategy's "prefer zero new dependencies" default |
| Works with the existing `ComponentActivity`/Compose setup without theme changes | Modest APK size increase (~1–2 MB; largely offset by already-present `androidx.activity`/`androidx.fragment` transitive deps) |

**Attempted and rejected — confirmed broken in practice, not just in theory.** B3 was implemented first (added `androidx.appcompat:appcompat:1.7.0`, the `AppLocalesMetadataHolderService` + `autoStoreLocales` meta-data, and a `Settings → Language` picker calling `AppCompatDelegate.setApplicationLocales()`). On-device testing (Galaxy S21 FE, API 35) showed the call **silently no-ops**: `AppCompatDelegate.getApplicationLocales()` read back empty immediately after `setApplicationLocales()`, with no exception anywhere in logcat. Calling the equivalent raw platform API (`context.getSystemService(LocaleManager::class.java).applicationLocales = ...`) in the same click handler worked immediately and correctly. Root cause: AppCompat's 33+ code path resolves an `Application`/`Activity` context internally via a hidden-API lookup that depends on `AppCompatActivity`'s lifecycle registration (or the `autoStoreLocales` shim registering equivalent callbacks) — on a plain `ComponentActivity` app this resolution silently fails rather than throwing. This is a real gap between AppCompat's documented "works without AppCompatActivity" claim and observed behavior, not a configuration mistake on this app's part (manifest merge was verified correct). **The `androidx.appcompat` dependency was removed entirely** once B4 (below) proved to work with zero dependencies.

### Option B4 (adopted): Direct platform `LocaleManager` (API 33+) + `Configuration`/`attachBaseContext` shim (API 26–32) — no dependency

Functionally the same runtime behavior as B2, but built and verified after B3's failure made B2's original "why hand-roll this" objection moot — there was no working zero-maintenance alternative. A small `common/AppLanguage.kt` object wraps:
- **API 33+:** `context.getSystemService(LocaleManager::class.java).applicationLocales` directly — confirmed working via on-device testing; the OS persists the choice and recreates activities automatically.
- **API 26–32:** the tag is persisted in `SharedPreferences` and applied via `Configuration.setLocale()` in `MyApplication.attachBaseContext()` and `MainActivity.attachBaseContext()`; the caller (`Settings` screen) calls `Activity.recreate()` after changing it, since no OS-level recreate mechanism exists below API 33.

| Pros | Cons |
|------|------|
| Zero new dependency; confirmed working on real hardware (not just API docs) | Two code paths (33+ vs. <33) to maintain, same as B2's original objection |
| No reliance on a third-party library's undocumented internal context resolution | The <33 path is untested on a physical device below API 33 in this pass (see Validation) |

### Decision: **B1 (system-locale fallback) + B4 (in-app switcher), both shipped in v1**; **B3 rejected with evidence**; **B2 superseded by B4**

**Rationale:** The original plan was B1-only for v1 with B3 as a v1.1 fast-follow. The user asked for the in-app switcher immediately, which surfaced B3's failure through real device testing rather than living undetected until a future release. Given B3 doesn't actually work as documented on this app's Activity shape, and B4 does — confirmed on hardware — B4 is strictly better than the original B2/B3 choice set: it has B3's zero-dependency property (better, even, since B3 turned out to need a dependency that didn't help) without B3's hidden failure mode. The two-code-path maintenance cost (B2's original objection) is accepted since it's the only verified-working option for API 26–32.

## Decision C — Non-Resource Content & Locale-Sensitive Formatting

### C1: `release_notes.yaml` stays English-only in v1

The file is auto-generated by CI from git commit log (English by convention; see `cee410e chore: exclude test/coverage commits from release notes generation`). Translating it would require either (a) a manual hand-translation step inserted into `release.yml`, breaking the "auto-generated" property, or (b) on-device/API machine translation, which violates the fully-offline NFR and adds a dependency. **Decision: leave `release_notes.yaml` English-only; the "What's New" section is supplementary, not core UX.** Revisit only if user feedback specifically requests it.

### C2: Force `Locale.US` for untranslated numeric/unit displays

`formatBytes()` (storage size) and the contrast percentage display (`"%.2f"`) render technical values paired with **untranslated** unit abbreviations (`GB`/`MB`/`KB`, `%`). Formatting these with `Locale.getDefault()` under `vi-VN` produces a mismatched style (comma decimal separator next to an English abbreviation, e.g. `1,5 GB`). **Decision: explicitly pass `Locale.US` to `String.format(...)` at these two call sites** (not the platform default), since the surrounding labels are not localized targets. This is a two-line fix bundled into Decision A's refactor commit, not a separate feature.

## Decision D — Regression Prevention

Android Lint ships with `MissingTranslation` and `ExtraTranslation` checks, enabled by default whenever more than one `values-*/` locale directory exists — no new dependency required. **Decision: rely on default Lint** (no custom lint config needed) and add `./gradlew lintDebug` as a required check alongside the existing `testDebugUnitTest` gate for any PR that touches `strings.xml`. This is documented as a new Quality Gate in Consequences below, not implemented by this ADR.

### Architecture / Data Flow

```
Device locale (Settings → Languages)
        │
        ▼
Android resource resolution
  values/strings.xml       (English, default — fallback)
  values-vi/strings.xml    (Vietnamese — new)
        │
        ▼
Composables call stringResource(R.string.x [, args])
  — replaces ~48 Text("literal") / contentDescription = "literal" call sites
        │
        ▼
Non-resource content (unaffected by locale):
  release_notes.yaml → English-only (Decision C1)
  formatBytes() / contrast display → Locale.US forced (Decision C2)
```

| Concern | Decision | APK Impact | Effort |
|---------|----------|------------|--------|

| New `values-vi/strings.xml` | — | < 10 KB | Translation of ~74 total strings/plurals |
| Locale switching | B1 + B4 | 0 MB | `common/AppLanguage.kt` (~45 LOC) + `attachBaseContext` overrides in `MyApplication`/`MainActivity` + Settings UI (~25 LOC) |

## Consequences

**Positive:**
- Resolves the open PRD question directly: v1 ships with full English + Vietnamese UI coverage, plus an in-app switcher independent of device locale.
- Zero new dependencies in the final state — B4 achieves what B3 promised (in-app switching) without the dependency B3 needed and without B3's hidden failure mode.
- Lint's built-in translation-completeness checks make future string additions self-enforcing once `values-vi/` exists.
- The B3 failure was caught by on-device testing *before* merging, not discovered later by a user — validates doing real-device QA rather than trusting library documentation alone for platform-integration features.

**Negative:**
- A1 is a one-time large-ish diff (9 files) — must be reviewed carefully for missed literals and for any test that asserts on hardcoded English text. One instance (`DocumentCard`'s page-count plural) was missed in the initial pass and only caught once a real scanned document existed to render it — the empty-state screens don't exercise this code path.
- `release_notes.yaml` stays English-only in *decision*, but its actual content today is already hardcoded Vietnamese — an accepted pre-existing inconsistency (Decision C1), not something this ADR fixes.
- The API 26–32 code path (`Configuration` override + manual `Activity.recreate()`) is unverified on a physical device in this pass — only the API 33+ (`LocaleManager`) path was confirmed on hardware (API 35).

**Risks:**
- `EditScreenTest.kt` and `DocumentListScreenTest.kt` currently match Compose nodes by hardcoded literal text (`onNodeWithText("...")`); after A1 these tests still pass against the English default resource, but are now indirectly coupled to `values/strings.xml` — a future contributor changing English copy without noticing the test coupling could break CI. Mitigate by preferring `onNodeWithText(context.getString(R.string.x))` over re-hardcoding the literal in the test.
- Format-string extraction (e.g. `"Page ${pageIndex + 1}"` → `%1$d`) is mechanical but error-prone (wrong positional index, or a translator reordering `%1$d`/`%2$s` incorrectly in `values-vi/`) — verify each interpolated string manually after translation.
- B4's API 26–32 shim is unverified on real hardware — if a contributor has access to an API 26–32 device/emulator, confirm `Activity.recreate()` correctly re-applies the wrapped `Configuration` before relying on it in production.
